# CloudPSS Skills V3 - Agent-First 设计方案

**文档版本**: 1.0  
**创建日期**: 2026-05-04  
**状态**: 设计阶段  

---

## 1. 设计背景与问题分析

### 1.1 当前问题

当前 V3 的 Portal 设计存在以下核心问题：

| 问题 | 表现 | 影响 |
|-----|------|------|
| **目标混乱** | 既想给 Agent 用，又想给人用 | 两者都照顾不好 |
| **信息过载** | 满屏 ID、路径、内部状态 | 用户看不懂，感到混乱 |
| **操作繁琐** | 创建 Case → Variant → Task → 执行 | 步骤太多，效率低 |
| **偏离专业** | 显示技术细节而非电力专业内容 | 电力工程师感到不专业 |
| **与 Agent 脱节** | 手动点击界面，Agent 无法参与 | 违背 Skills 设计的初衷 |

### 1.2 核心洞察

> **用户的真实需求**：不是需要一个"操作界面"，而是需要一个"能帮我完成复杂科研工作的智能助手"。

**关键转变**：
- 从 **"人操作界面"** → 改为 **"Agent 执行，人监督干预"**
- 从 **"功能展示"** → 改为 **"目标驱动"**
- 从 **"技术透明"** → 改为 **"专业语言"**

## 2. 新设计定位

### 2.1 产品定位

**一句话定义**：
> CloudPSS Skills V3 是**面向电力系统科研的 Agent 基础设施**，让 AI Agent 能够理解、调用、组合电力仿真技能，辅助研究人员完成复杂科研工作。

### 2.2 架构分层

```
┌─────────────────────────────────────────────────────────────┐
│  用户 (电力研究人员)                                         │
│     ↓ 自然语言指令                                           │
├─────────────────────────────────────────────────────────────┤
│  Agent (Claude/Copilot/Gemini)                              │
│     ↓ MCP Protocol                                          │
├─────────────────────────────────────────────────────────────┤
│  CloudPSS MCP Server                                        │
│  ┌──────────────┬──────────────┬──────────────┐            │
│  │   Discovery  │    Tools     │  Resources   │            │
│  │   (技能发现)  │   (工具调用)  │   (资源访问)  │            │
│  └──────────────┴──────────────┴──────────────┘            │
│     ↓ 调用                                                   │
├─────────────────────────────────────────────────────────────┤
│  CloudPSS Skills API                                        │
│  ┌─────────┬─────────┬─────────┬─────────┬─────────┐       │
│  │  Server │   Case  │ Variant │   Task  │  Result │       │
│  │  服务器  │   案例  │   变体   │   任务  │   结果   │       │
│  └─────────┴─────────┴─────────┴─────────┴─────────┘       │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 Portal 新定位

**从**：电力仿真业务管理平台  
**改为**：**Agent 执行观测站**

| 维度 | 旧定位 | 新定位 |
|-----|--------|--------|
| **主要用户** | 人（手动操作） | Agent（自动执行） |
| **人的角色** | 操作者 | 监督者、决策者 |
| **核心功能** | CRUD 管理 | 状态观测、异常处理 |
| **使用频率** | 每次都用 | 仅在需要查看/干预时用 |
| **信息展示** | 所有细节 | 关键状态、异常告警 |

## 3. 核心设计原则

### 3.1 Agent-First 原则

1. **所有功能 API 化**：Agent 能通过 MCP 调用所有能力
2. **异步执行**：Agent 提交任务后可做其他事，无需阻塞等待
3. **状态透明**：Agent 能随时查询任务进度和结果
4. **中断协作**：需要人决策时，Agent 能请求确认

### 3.2 专业语言原则

1. **说电力工程师的话**：母线、支路、潮流、暂态，而非 ID、路径、JSON
2. **隐藏技术细节**：用户不需要知道内部数据结构
3. **业务语义优先**：显示"IEEE39 系统"而非 `case_20250504_abc123`

### 3.3 分层观测原则

| 层级 | 内容 | 使用场景 |
|-----|------|---------|
| **概览层** | 当前在做什么、进度、预计完成时间 | 日常监控 |
| **详情层** | 具体参数、中间结果、日志 | 问题排查 |
| **控制层** | 暂停、取消、重试、参数调整 | 异常干预 |

### 6.1 核心原则

1. **只读为主**：人看但不操作
2. **关键优先**：只显示最重要的信息
3. **异常突出**：需要干预时才醒目提示
4. **专业语言**：用电力术语而非技术术语

## 8. 关键决策记录

| 决策 | 选择 | 理由 |
|-----|------|------|
| **协议选择** | MCP | 开放标准，Claude 原生支持 |
| **Portal 定位** | 观测站而非操作台 | Agent 是执行主体 |
| **信息展示** | 专业语言 > 技术细节 | 用户是电力工程师 |
| **交互模式** | 异步为主 | 科研任务耗时长 |
| **异常处理** | Agent 请求确认 | 关键决策需人参与 |
