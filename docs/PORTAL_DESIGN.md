版本: v1.0  
日期: 2026-05-02

---

## 1. 定位与目标

### 1.1 核心定位

CloudPSS 收纳大师 Portal 是面向**电力系统研究者**的仿真研究工作台，不是 CLI 的网页壳，也不是简单的任务列表管理器。

核心目标：让用户对**"模型、参数、仿真配置、执行过程、结果、分析、报告"**形成全过程掌控。

### 1.2 用户核心问题

用户进入 Portal 后，应能清楚回答：

- [x] 我现在研究的是哪个系统模型？
- [x] 模型里有哪些元件？每个元件参数是什么？
- [x] 我要算什么？潮流、EMT、稳定性？
- [x] 仿真参数是什么？步长、时长、输出通道？
- [x] 任务是否提交、运行、完成、失败？
- [x] 结果是什么？表格、曲线、通道在哪里？
- [ ] 报告是否把模型、配置、结果都汇总清楚？
- [x] 所有操作有没有审计记录和可追溯证据？

Portal 主导航围绕**研究流程**组织：

### 3.2 模型与算例 (Cases) - 核心页面

**目标**：成为整个 Portal 的主工作台，让用户完全掌控模型。

#### 3.2.1 当前实现

**布局**：左右分栏
- 左侧：Case 树（名称、RID、Tasks 数量）
- 右侧：详情面板

**模型编辑器**：
- 按 definition 自动分组（Bus/Line/Transformer/Generator/Load/Control）
- 表格展示：元件名称、definition、canvas、args 参数
- 单元格直接编辑
- dirty 状态标记
- 保存前自动备份（`.bak-{timestamp}`）

**远端模型提示**：
```
当前 Case 使用远端 RID。请先在任务中配置本地 model_source，
Portal 才能表格化编辑模型文件。
```

#### 3.2.3 模型版本与历史

**需求**：追踪模型的每一次修改。

**实现方案**：
```yaml
# .cloudpss/cases/{case_id}/history/
case.yaml                    # 当前模型
history/
  ├── 20250502_143052.yaml  # 自动备份
  ├── 20250502_150123.yaml
  └── manifest.json         # 历史记录索引
```

**功能**：
- 每次保存自动生成备份
- 历史列表显示：时间、修改摘要、操作人
- 对比视图：并排显示两个版本的差异
- 回退功能：恢复到指定版本

## 4. 关键设计原则

Portal 的设计核心不是"按钮齐全"，而是**"研究者掌控感"**。

### 4.1 已实现原则

| 原则 | 实现状态 |
|------|----------|
| 模型参数表格化，不让用户读 YAML | ✅ 已实现 |
| 结果曲线表格化，不让用户翻 artifacts | ✅ 已实现 |
| 所有修改必须可追踪 | ⚠️ 部分实现（有审计，无版本历史） |
| 不能做的事情明确说明 | ✅ 已实现（远端模型提示） |

### 4.2 待实现原则

| 原则 | 目标状态 |
|------|----------|
| 仿真配置结构化，不让用户猜 JSON | ❌ 需实现表单化配置 |
| 报告必须自动引用模型、配置、任务、结果 | ⚠️ 需增强报告内容 |
| 所有修改可备份、可回退 | ⚠️ 需实现版本历史 |

## 7. 总结

当前 Portal 已实现**基础工作流**：
- Case → Task → Result 的完整链路
- 模型参数表格化编辑
- 任务运行前检查
- 结果可视化和审计记录

距离**理想状态**的差距主要在：
1. 仿真配置仍需 JSON 编辑（需表单化）
2. 模型页面布局偏向开发者工具（需重构为工作台）
3. 结果分析功能较基础（需增强可视化）
4. 缺少版本历史（需实现备份管理）

建议按 P0 → P1 → P2 优先级逐步完善。
