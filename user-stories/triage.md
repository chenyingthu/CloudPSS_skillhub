# 遗产 Stories 筛选结论

本文件回答两个问题：

1. 从遗产 stories 里有什么值得借鉴
2. 这些内容怎样才能被当前仓库安全吸收

## 一、可直接借鉴

这些方向和当前仓库主线高度一致，只需要改写成当前仓库语言即可：

### 1. 研究起点与工作副本

- 遗产来源：
  `legacy/01-model-building.md`
- 可借鉴点：
  先从模板或已知算例起步，再创建研究分支继续改模
- 当前吸收方式：
  对齐到 `Model.fetch / Model.load / Model.dump / model.save`
- 不该保留的部分：
  “复制公开算例到个人空间”的产品式措辞，以及不存在于当前仓库的 `manage.copyModel()`

### 2. 多场景潮流研究

- 遗产来源：
  `legacy/02-simulation.md`
  `legacy/04-batch-optimization.md`
- 可借鉴点：
  批量比较多工况、整理关键指标、生成结论先行摘要
- 当前吸收方式：
  对齐到 `powerflow_engineering_study_example.py` 与 `powerflow_batch_study_example.py`
- 不该保留的部分：
  泛化成“任意批量优化平台”或无证据的优化求解宣称

### 3. 安全筛查与严重性排序

- 遗产来源：
  `legacy/03-security-analysis.md`
- 可借鉴点：
  “扫描范围 -> 排序 -> 薄弱环节 -> 报告”的结构化研究输出
- 当前吸收方式：
  保留到 IEEE39 潮流 N-1 和 IEEE3 EMT N-1 两条已验证路径
- 不该保留的部分：
  对发电机 N-1、N-2、通用全网安全平台的外推

### 4. 结果导出与报告

- 遗产来源：
  `legacy/05-reporting.md`
- 可借鉴点：
  将研究结果整理为适合复核和汇报的输出件
- 当前吸收方式：
  保留 Markdown/CSV 方向，继续强调结论、表格和证据边界
- 不该保留的部分：
  直接承诺 Word/PDF/PPT 自动化产物

## 二、可改写后保留

这些方向不是当前主线已完成能力，但经过收紧后可以作为 future backlog：

### 1. 研究摘要与汇报材料分层

- 遗产来源：
  `legacy/05-reporting.md`
- 为什么可保留：
  当前仓库已经有 Markdown 报告和 CSV 导出，再往前一步是统一摘要模板，而不是做办公文档平台
- 当前限制：
  只应做 repo 内可验证的 Markdown/表格/文本 digest，不优先做 Word/PDF/PPT

### 2. 参数化研究矩阵

- 遗产来源：
  `legacy/02-simulation.md`
  `legacy/04-batch-optimization.md`
- 为什么可保留：
  当前已经有潮流批量研究、EMT 严重度扫描、EMT 清除时间扫描
- 当前限制：
  只能围绕现有 IEEE39/IEEE3 已验证参数维度继续扩，不能直接宣称通用优化框架

## 三、当前应明确淘汰

这些内容不应进入当前项目下一阶段重点：

### 1. 另一套 skills 产品接口

- 遗产问题：
  `n1Analysis.*`、`batchEnhanced.*`、`report.*`、`market.*` 等命名并不对应当前仓库真实接口
- 处理结论：
  全部退回为历史背景，不再作为当前文档用语

### 2. 与当前主线无关的扩展方向

- 遗产问题：
  市场、碳排放、微电网、HVDC、配网重构、新能源消纳等大量方向超出当前路线
- 处理结论：
  当前不纳入 backlog

### 3. 无 live 证据的“稳定/已实现”标签

- 遗产问题：
  遗产 stories 中大量状态判定与当前仓库代码、测试、探针记录矛盾
- 处理结论：
  当前项目只认仓库内真实脚本、测试与 live 记录，不继承这些标签

## 四、对当前项目真正有价值的后续方向

如果要从遗产中提炼出对当前项目有价值的进一步发展，优先级建议如下：

1. 若未来再新增研究型 story，必须先落到现有矩阵中的相邻维度，再考虑是否值得扩成新能力
2. 把案例库、验收清单和证据登记表继续保持同步，避免文档等级漂移
3. 在不引入办公文档平台化的前提下，统一当前 Markdown/CSV/digest 输出模板

这些方向都能借鉴遗产 stories 的“研究问题驱动”表达方式，但不会把仓库重新带回一个没有证据约束的大而全需求池。
